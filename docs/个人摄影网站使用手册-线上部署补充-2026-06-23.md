## 线上部署与首页发布图

### 线上地址

- 线上网站：<https://cartonmouse.github.io/camerasnowsong/>
- GitHub 仓库：<https://github.com/cartonmouse/camerasnowsong>
- 部署状态：<https://github.com/cartonmouse/camerasnowsong/actions>

网站通过 GitHub Actions 自动部署到 GitHub Pages。推送到 `main` 分支后，会自动触发部署工作流。

常规上传流程：

```powershell
git status
git add .
git commit -m "本次模块说明"
git push
```

### 本地构建检查

普通本地构建：

```powershell
npm.cmd run build
```

GitHub Pages 路径模式构建：

```powershell
$env:DEPLOY_TARGET='github-pages'
npm.cmd run build
Remove-Item Env:\DEPLOY_TARGET
```

GitHub Pages 是项目页，线上路径带有 `/camerasnowsong/`。因此部署模式下 Vite 的 `base` 会设置为 `/camerasnowsong/`。

### 首页发布图

当前首页不是直接发布原图，而是使用压缩后的 WebP 发布图：

- 发布图目录：`public/site-photos/home/`
- 当前数量：9 张首页 Star 图片
- 主站数据：`src/data/photos.json`
- 当前首页的 `photo.src` 已指向 `/site-photos/home/*.webp`

注意：

- 不建议把大量原图、RAW、CR2 或未经压缩的大图直接提交到 GitHub。
- 当前首页点开后的大图和列表图使用同一个 `photo.src`，也就是同一张压缩 WebP。
- 目前还没有正式区分 `thumb` 缩略图和 `src` 大图。
- 如果运行 `npm.cmd run photos:sync`，可能会重新生成 `src/data/photos.json`，从而覆盖首页发布图路径。运行后需要检查线上图片路径。

### 线上图片破图排查

如果网站能打开但图片破图，优先检查：

1. 图片文件是否真的提交到了 GitHub。
2. 图片路径是否适配 `/camerasnowsong/` 子路径。
3. GitHub Actions 是否已经完成最新一次部署。
4. `src/data/photos.json` 是否又被 `photos:sync` 覆盖回 `/photos/...`。
5. 浏览器是否缓存了旧页面，可以强制刷新后再看。

目前图片组件会给所有以 `/` 开头的 `photo.src` 自动加上 Vite 的 `BASE_URL`，因此 `/photos/...` 和 `/site-photos/...` 都应该能适配 GitHub Pages。

### 长期图片方案

当前方案适合第一阶段上线，不适合长期管理大量作品图。

后续建议建立正式发布流程：

- 从本地原图自动生成 `thumb` 和 `large` 两套 WebP。
- 列表页加载 `thumb`，Lightbox 加载 `large`。
- 管理后台或 npm 脚本自动维护发布图路径。
- 如果作品数量继续增加，再考虑对象存储或 CDN。
