# Local Album Admin Design

## Goal

Build a local-only album management tool for the personal photography website so new photo folders can be classified and described through a browser UI instead of manually editing `album.json`.

The first version manages album-level metadata only:

- `title`
- `topics`
- `description`
- `featured`

It does not manage single-photo metadata, uploads, deletion, login, public hosting, databases, cover selection, or sorting.

## Current Context

The project is currently a static photography portfolio.

Project path:

```text
D:\3BUPT\mark'workshop\personal-photography-website-superpowers
```

Photos live under:

```text
public/photos/
```

Each first-level folder is treated as an album:

```text
public/photos/沙金/
public/photos/1.18星见雅（已发）/
public/photos/2知更鸟_音乐台/
```

Album metadata is stored in:

```text
public/photos/<album-folder>/album.json
```

Website display data is generated by:

```bash
npm.cmd run photos:sync
```

Current frontend preview is served by Vite:

```text
http://localhost:5173/
```

The proposed admin tool will be local-only and served separately:

```text
http://localhost:4000/admin
```

## Why Local-Only

A static website can display prepared files, but it cannot safely write local files such as `album.json`.

The local admin tool solves that by running a Node service on the user's computer. The browser UI talks to this local service, and the service can read and write project files.

This keeps the public-facing website simple and static while giving the user a comfortable local management workflow.

## MVP Workflow

1. User manually places a new folder under `public/photos/`.
2. User starts the local admin tool.
3. User opens `http://localhost:4000/admin`.
4. Admin page lists albums and marks incomplete albums.
5. User selects an album.
6. User edits album-level fields:
   - title
   - topics
   - description
   - featured
7. User clicks save.
8. Local service writes `album.json`.
9. User clicks sync.
10. Local service regenerates `src/data/photos.json`.
11. User refreshes the public portfolio at `http://localhost:5173/`.

## Scope

### In Scope

- Start a local admin server with an npm script, likely:

```bash
npm.cmd run admin
```

- Serve an admin page at:

```text
http://localhost:4000/admin
```

- Scan `public/photos/` for first-level album folders.
- Count image files in each album.
- Read existing `album.json` when present.
- Create or update `album.json` when the user saves.
- Display album status:
  - configured
  - missing metadata
  - missing topics
  - missing description
- Provide a fixed topic multi-select list:
  - 风景
  - 人像
  - Cosplay
  - 城市
  - 旅行
  - 舞台
  - 纪实
  - 日常
  - 待分类
- Allow entering custom topics if needed.
- Show a few image previews for the selected album.
- Trigger the existing photo sync logic after saving or through a separate sync button.
- Preserve existing `photos` single-photo overrides when saving album-level fields.

### Out of Scope

- Uploading images.
- Deleting images.
- Moving or renaming image files.
- Editing single-photo title, description, location, topics, or featured state.
- Choosing album cover.
- Sorting albums or photos.
- Login, permissions, or public admin deployment.
- Database storage.
- AI-based image classification.

## Data Model

The admin tool continues using `album.json` as the source of truth.

Example:

```json
{
  "title": "1.18星见雅（已发）",
  "topics": ["Cosplay", "人像"],
  "description": "一组关于星见雅角色造型的照片。",
  "featured": true,
  "photos": {}
}
```

When saving, the service must preserve fields it does not manage, especially:

```json
{
  "photos": {
    "IMG_0661-已增强-NR.jpg": {
      "title": "大选获胜！",
      "description": "在新的竞选当中，我胜利了！"
    }
  }
}
```

That means album-level save should merge managed fields into the existing JSON rather than replacing the whole file with a minimal object.

## Server Design

Create a local Node server, for example:

```text
scripts/admin-server.mjs
```

Responsibilities:

- Serve static admin HTML/CSS/JS.
- Provide JSON APIs.
- Read and write `album.json`.
- Reuse or call existing photo sync code.

Proposed endpoints:

```text
GET  /admin
GET  /api/albums
GET  /api/albums/:albumId
PUT  /api/albums/:albumId
POST /api/sync
```

### GET /api/albums

Returns a list like:

```json
[
  {
    "id": "1.18星见雅（已发）",
    "title": "1.18星见雅（已发）",
    "topics": ["Cosplay", "人像"],
    "description": "一组关于星见雅角色造型的照片。",
    "featured": true,
    "imageCount": 24,
    "status": "configured",
    "previewImages": [
      "/photos/1.18%E6%98%9F%E8%A7%81%E9%9B%85%EF%BC%88%E5%B7%B2%E5%8F%91%EF%BC%89/IMG_0001.jpg"
    ]
  }
]
```

### PUT /api/albums/:albumId

Accepts:

```json
{
  "title": "1.18星见雅（已发）",
  "topics": ["Cosplay", "人像"],
  "description": "一组关于星见雅角色造型的照片。",
  "featured": true
}
```

Writes or updates:

```text
public/photos/<albumId>/album.json
```

### POST /api/sync

Runs the same behavior as:

```bash
npm.cmd run photos:sync
```

It should return:

```json
{
  "ok": true,
  "message": "Synced 81 photos"
}
```

## Admin UI Design

The admin page should be compact and work-focused, not a marketing page.

Layout:

- Left column: album list.
- Right panel: selected album editor.
- Top bar: page title and sync button.

Album list item should show:

- album title or folder name
- image count
- status label
- topics summary

Editor should show:

- title text input
- topics multi-select as checkboxes or pill toggles
- custom topic input
- description textarea
- featured checkbox
- preview image strip
- save button
- save status message

## Status Rules

An album can be marked:

```text
missing-config
```

No `album.json` exists.

```text
needs-topics
```

`topics` is empty or missing.

```text
needs-description
```

`description` is empty or missing.

```text
configured
```

Title, topics, and description are present.

These labels are only for admin visibility. They do not change public website behavior by themselves.

## Error Handling

The admin service should handle:

- invalid JSON in existing `album.json`
- missing album folder
- failed file write
- failed sync
- no image files in a folder

For invalid `album.json`, the API should not silently overwrite the file. It should return an error and ask the user to fix or back up the file first.

## Security Boundary

This tool is local-only.

It should bind to:

```text
127.0.0.1
```

not a public network interface.

It should not include public deployment instructions in the first version.

## Testing Plan

Automated tests should cover the server-side file logic:

- scan albums from a temporary `public/photos` fixture
- detect missing `album.json`
- read existing `album.json`
- preserve unmanaged fields when saving
- write valid UTF-8 JSON with Chinese text
- reject invalid album IDs that try path traversal
- sync endpoint can call sync helper or be tested through a stubbed function

Manual verification should cover:

- start admin server
- open `http://localhost:4000/admin`
- edit an album
- save
- inspect `album.json`
- run sync from UI
- refresh public website and confirm topics are visible

## Acceptance Criteria

- `npm.cmd run admin` starts a local admin server.
- `http://localhost:4000/admin` loads successfully.
- The page lists all first-level folders under `public/photos`.
- Albums without `album.json` are visible and editable.
- User can edit title, topics, description, and featured.
- Save creates or updates `album.json`.
- Existing `photos` overrides are preserved.
- Sync regenerates `src/data/photos.json`.
- Public website shows newly classified albums under selected topics.
- Tests pass with `npm.cmd run test:data` or a new admin test command.
