# 个人摄影网站 - 上线部署阶段总结 - 2026-06-23

今天完成了个人摄影网站的第一阶段正式上线部署。

- 线上网站：<https://cartonmouse.github.io/camerasnowsong/>
- GitHub 仓库：<https://github.com/cartonmouse/camerasnowsong>
- 部署方式：GitHub Actions 自动构建并发布到 GitHub Pages
- 当前可见结果：首页已经能正常显示压缩后的发布图
- 当前限制：目前只处理了首页展示用图片，作品页的大量照片发布流程还没有正式建立

## 今天完成的工作

### 1. 简化“关于”页面

“关于”页面从早先偏完整的文字型页面，改成了极简版本：

```text
Amor Fati

paperboxmouse@163.com
通过这个邮箱可以联系我
```

这次调整的目的不是继续扩写摄影宣言，而是把页面降到一个克制、清楚、可上线的状态。页面只保留一个精神锚点和联系方式。

### 2. 建立 GitHub Pages 部署流程

项目仍然是 React + Vite 主站。为了适配 GitHub Pages 的项目页路径，新增了 GitHub Pages 部署配置：

- `vite.config.js` 根据 `DEPLOY_TARGET=github-pages` 把 `base` 设置为 `/camerasnowsong/`
- `.github/workflows/deploy.yml` 负责在 `main` 分支 push 后自动构建并发布
- 工作流包含安装依赖、数据测试、管理工具测试、构建、上传 Pages artifact、部署 Pages

这意味着后续正常流程是：

```powershell
git status
git add .
git commit -m "本次模块说明"
git push
```

push 到 GitHub 后，GitHub Actions 会自动部署。部署状态可以在仓库的 Actions 页面查看。

### 3. 处理 GitHub Pages 首次部署限制

一开始进入 Pages 设置页时，GitHub 提示需要升级或公开仓库才能启用 Pages。原因是免费 GitHub Pages 对私有仓库有访问限制。

解决方式：

- 将仓库作为公开展示项目处理
- 在 Settings -> Pages 中选择 GitHub Actions 作为部署来源
- 通过 push 触发自动部署

这里也确认了一个原则：只要图片要在公开网站显示，它本质上就必须是公开可访问的。区别只在于是否把原图直接放进 Git 仓库，以及是否采用对象存储、CDN、压缩图、访问控制等更适合长期维护的方式。

### 4. 修复 GitHub Pages 下的资源路径问题

第一次上线后，页面能打开，但图片是破图。这里遇到了两类问题。

第一类是 Vite 项目页路径问题：

- 本地网站运行在 `/`
- GitHub Pages 项目页运行在 `/camerasnowsong/`
- 如果图片路径写成 `/photos/...`，浏览器会去访问 `https://cartonmouse.github.io/photos/...`
- 正确访问应该是 `https://cartonmouse.github.io/camerasnowsong/photos/...`

解决方式是在图片组件中统一处理根路径资源：

- 如果 `photo.src` 以 `/` 开头，就自动加上 `import.meta.env.BASE_URL`
- 这样 `/photos/...`、`/site-photos/...` 这类路径都能适配 GitHub Pages 的子路径

第二类是图片文件本身没有被部署：

- 项目 `.gitignore` 忽略了大量图片格式
- 原始照片没有进入 GitHub 仓库
- GitHub Pages 上自然访问不到这些文件

这说明“代码已部署”和“图片已发布”是两件事。

### 5. 生成首页发布图

为了不把大体积原图直接塞进 Git 仓库，今天采用了短期方案：

- 只处理首页当前展示的 Star 图片
- 从本地原始照片生成压缩 WebP 发布图
- 输出到 `public/site-photos/home/`
- 更新 `src/data/photos.json` 中首页 9 张图的路径

本次处理结果：

- 首页发布图数量：9 张
- 原始首页图总体积约：58.9 MB
- 发布 WebP 总体积约：1.67 MB
- 输出路径：`public/site-photos/home/`

这不是最终照片资产方案，但足够支撑首页第一阶段上线。

### 6. 修复 `/site-photos/` 路径破图

生成首页发布图后，部署再次成功，但首页仍然破图。原因是最初的路径修复只覆盖了 `/photos/`，而新的发布图路径是 `/site-photos/`。

解决方式：

- 将图片组件逻辑改成处理所有以 `/` 开头的根路径
- 不再只判断 `/photos/`
- 再次部署后，首页图片正常显示

### 7. 明确当前 Lightbox 行为

用户确认首页图能显示后，又确认“点开后展示的也是缩略图吗”。

当前结论：

- 是的，目前缩略图和点开后的大图使用的是同一个 `photo.src`
- 因此首页点开后展示的也是那张压缩 WebP 发布图
- 当前还没有建立 `thumb` 和 `full` 两套图片字段

后续如果想让列表加载快、点开后更清晰，需要把数据结构扩展成类似：

```json
{
  "thumb": "/site-photos/home/xxx-thumb.webp",
  "src": "/site-photos/home/xxx-large.webp"
}
```

然后让列表使用 `thumb`，Lightbox 使用 `src`。

## 遇到的问题与解决方法

### 问题 1：GitHub Pages 页面提示需要升级或公开仓库

原因：免费 GitHub Pages 对私有仓库发布有限制。

解决：将仓库作为公开展示项目处理，并使用 GitHub Actions 发布。

### 问题 2：部署成功但图片破图

原因包括：

- 图片文件没有进入 GitHub 仓库
- GitHub Pages 项目页有 `/camerasnowsong/` 子路径
- 代码中的根路径资源没有自动加上 Vite 的 `BASE_URL`

解决：

- 生成少量首页发布图并提交
- 修改图片组件，给根路径资源统一加上 `BASE_URL`

### 问题 3：不适合把大量原图直接上传 GitHub

原因：

- Git 仓库不适合长期管理大量大体积图片
- 原图会让仓库快速膨胀
- 后续替换、压缩、缓存、迁移都不方便

短期解决：

- 只发布首页需要的压缩 WebP
- 原始照片继续保留在本地 `public/photos/`

长期方向：

- 使用对象存储或 CDN 管理正式图片资产
- 或建立自动生成发布图的脚本和后台流程

### 问题 4：`photos:sync` 可能覆盖发布图路径

当前 `src/data/photos.json` 是主站实际读取的数据。首页发布图路径已经写入这个文件，但如果再次运行 `npm.cmd run photos:sync`，有可能从 `public/photos/` 和 `album.json` 重新生成数据，覆盖这些手动发布路径。

临时注意：

- 运行 `photos:sync` 后需要检查首页图片路径
- 必要时重新生成首页发布图并更新 `src/data/photos.json`

长期方向：

- 把发布图生成纳入脚本
- 让管理后台或同步脚本自动维护 `thumb/src` 发布路径

## 流程反思

这次中途使用了较重的开发流程，包含更完整的计划、检查、部署排错和反复验证。对上线部署这种跨本地、GitHub、Actions、Pages、图片资源路径的问题来说，这种谨慎有价值；但对小修改来说会显得耗时、耗 token。

后续约定：

- 小改动采用轻量模式：先读关键文件，直接改，跑必要测试，简短总结
- 大模块或疑难问题再使用完整流程
- 每完成一个模块提醒提交并上传 GitHub
